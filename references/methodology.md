## 1. Epistemic Ledger

Before deriving a model, classify important statements:

| Label | Meaning | Requirement |
|---|---|---|
| Evidence | Externally supported observation or finding | Cite it in Research mode |
| Given | User-supplied or explicitly accepted fact | Name its source |
| Assumption | Provisional premise | Explain why it is reasonable |
| Inference | Conclusion derived from prior statements | Show the reasoning link |
| Synthesis | New language, model, or framework | Do not disguise it as consensus |

Do not call a premise "first-principles" merely because it sounds fundamental.
A useful premise must be relevant, independently defensible, and hard to remove
without making the target phenomenon unintelligible.

Different premise types have different stability:

- physical and mathematical constraints are usually highly stable;
- biological and cognitive constraints are population- and context-sensitive;
- economic constraints depend on scarcity, incentives, and institutions;
- social and organizational constraints can change when norms or rules change;
- technical constraints can become obsolete quickly.

## 2. Ten-Stage Derivation

### Stage 1: Set the Object

**Purpose:** Prevent level confusion.

Ask:

- What exactly is being analyzed?
- Is it a capability, process, role, product, institution, outcome, or label?
- At what level: person, team, organization, market, or society?
- In which context and time horizon?
- What decision or use should the answer support?
- What is explicitly out of scope?

**Output:** One-sentence object statement plus exclusions.

**Common mistake:** Switching between an individual trait and an organizational
system without noticing.

### Stage 2: Suspend Inherited Labels

**Purpose:** Avoid circular explanation.

List familiar definitions, analogies, formulas, and frameworks as candidates.
Do not use them as premises until their components and causal relevance have
been justified.

**Output:** A short list of inherited assumptions being held provisionally.

**Common mistake:** "X is a kind of Y" followed by explaining Y with another
label.

### Stage 3: Find the Irreducible Problem

**Purpose:** Explain why the object exists.

Ask:

- What recurring problem, tension, or coordination need does it solve?
- What would still need to be accomplished if the current label disappeared?
- In a world without this object, what predictable cost or failure appears?
- Is the object necessary, or only one historical solution?

**Output:** Need -> consequence without solution -> function performed.

**Common mistake:** Confusing the current implementation with the underlying
function.

### Stage 4: Expose Constraints

**Purpose:** Identify what any viable solution must respect.

Check:

- scarcity and opportunity cost;
- uncertainty and information cost;
- conflicting goals or stakeholders;
- time, reversibility, and path dependence;
- incentives, power, and responsibility;
- capacity and attention limits;
- delayed, noisy, or biased feedback;
- technical, legal, ethical, and institutional boundaries.

**Output:** Ranked constraints, each labeled by type and stability.

**Common mistake:** Treating a current convention as an immutable constraint.

### Stage 5: Derive the Mechanism

**Purpose:** Replace association with causality.

Build the shortest adequate chain:

```text
inputs -> transformation/mechanism -> immediate output -> downstream outcome
```

For each arrow ask:

- What causes this transition?
- Under what condition does it occur?
- What competing cause could produce the same result?
- What feedback loop or actor adaptation changes the pathway?

**Output:** Causal chain with assumptions and missing evidence marked.

**Common mistake:** Jumping from a feature directly to a business or human
outcome.

### Stage 6: Build the Minimum Sufficient System

**Purpose:** Avoid arbitrary lists.

For every proposed component:

1. Define its unique function.
2. Remove it mentally.
3. State the specific failure produced by removal.
4. Check whether another component already performs the same function.

Then test sufficiency:

- If all components exist, can the claimed function still fail?
- What enabling condition was omitted?
- Which component is core, and which merely improves performance?

Classify components as:

- necessary;
- conditionally necessary;
- enabling;
- optional enhancer;
- outcome rather than component.

**Output:** Minimum system plus deletion and sufficiency results.

**Common mistake:** Calling every correlated factor a core component.

### Stage 7: Draw Boundaries

**Purpose:** Make the concept discriminating.

Compare with 2-5 neighboring concepts using common dimensions:

- function;
- object of action;
- mechanism;
- decision rights or responsibility;
- time horizon;
- observable output;
- failure signature.

Also separate:

```text
inputs | enabling conditions | core mechanism | outputs | outcomes
```

**Output:** Boundary table and at least one positive and negative example.

**Common mistake:** A definition broad enough to include everything.

### Stage 8: Stress-Test

**Purpose:** Find where the model breaks.

Ask:

- What is the strongest counterexample?
- What alternative model explains the same observations?
- Which stakeholder would disagree, and why?
- What happens at extreme scale, low frequency, high uncertainty, or hostile
  incentives?
- Which claimed benefit creates a second-order cost?

**Output:** Counterexample, alternative explanation, and failure boundary.

**Common mistake:** Listing generic "risks" without challenging the model.

### Stage 9: Operationalize

**Purpose:** Make the model observable and corrigible.

Define:

- direct behaviors or events;
- proxy indicators and their limitations;
- leading and lagging measures;
- baseline and comparison;
- predicted pattern if the model is true;
- observation that would weaken or disprove it.

Do not confuse an easy-to-measure proxy with the construct.

**Output:** Indicator table and falsification condition.

**Common mistake:** Using the desired outcome as proof that the mechanism
existed.

### Stage 10: Translate to Use

**Purpose:** Make the derivation consequential.

Derive implications rather than appending generic advice:

- What should the user stop assuming?
- What question should be asked first?
- What design or decision changes?
- What small test buys the most information?
- What remains unknowable or context-dependent?

**Output:** 3-5 implications directly linked to prior mechanisms.

**Common mistake:** Recommendations that could have been written before the
analysis.

## 3. Required Tests

### Counterfactual Test

Remove the object or current solution. Does the underlying need remain? If not,
the proposed function may be invented.

### Deletion Test

Remove one component. Name the unique failure. If no unique failure appears,
the component may be redundant or optional.

### Sufficiency Test

Assume every proposed component is present. Can the claimed function still
fail? If yes, identify the missing condition.

### Causal-Chain Test

For every arrow, name the mechanism and a competing cause. Unexplained arrows
are hypotheses, not conclusions.

### Boundary Test

Find a case that satisfies the definition but should not belong, and a case
that should belong but fails the definition. Revise until both errors shrink.

### Alternative-Explanation Test

State the strongest competing model and what evidence would discriminate
between it and the preferred model.

### Falsification Test

State an observable result that would materially lower confidence in the model.
"More research is needed" is not a falsification condition.

## 4. Pseudo-First-Principles Moves

Reject or relabel these moves:

- A famous quote is not a premise.
- An analogy generates hypotheses; it does not prove mechanisms.
- A catchy formula is a heuristic unless formally justified.
- "Humans naturally..." requires a population, context, and evidence.
- "Technology always..." usually hides historical contingencies.
- Decomposing a word into synonyms is not decomposition of the phenomenon.
- A long component list is not a minimum system.
- A successful outcome does not prove a high-quality process.
